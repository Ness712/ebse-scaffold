# Issue Tracking

**[CHOIX D'EQUIPE]** Linear (satisfaction dev) ou GitHub Issues (simplicite). Kanban 4 colonnes | Score GRADE : 2/7

## Outils

| Outil | Forces | Limites |
|-------|--------|---------|
| Linear | UX rapide, satisfaction dev elevee | Payant, ecosysteme ferme |
| GitHub Issues | Integre au code, gratuit, Projects v2 | UX basique pour gros volumes |
| Jira | Standard entreprise, puissant | Complexe, lent, sur-configuration |

## Kanban recommande

```
Backlog → In Progress → Review → Done
```

Limiter le WIP (Work In Progress) a 2-3 items par dev. Chaque issue a un owner clair et des criteres d'acceptation.

## Regles

| Regle | Source |
|-------|--------|
| 1 issue = 1 probleme ou 1 feature | SWEBOK v4 — tracabilite |
| Labels pour categoriser (bug, feat, chore) | SWEBOK v4 — triage efficace |
| Triage hebdomadaire du backlog | SWEBOK v4 ch.9 — gestion de projet |

Sources : SO Developer Survey satisfaction (niv. 5), SWEBOK v4 ch.9 gestion de projet (niv. 5)
